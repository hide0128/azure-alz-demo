# Control 3.4 Landing Zone CI/CDデモ 完全ガイド

## このデモって何をするの？

このデモは、**Azureにネットワークを作る作業を自動化する仕組み**を見せるものです。

### たとえ話で説明すると...

学校で図工の作品を作ることを想像してください：

- **Bicepファイル** = 設計図（どんな作品を作るか書いた紙）
- **GitHub** = みんなで設計図を共有する場所
- **CI（自動チェック）** = 先生が設計図をチェックして「これで大丈夫！」と言ってくれる
- **CD（自動デプロイ）** = 設計図通りに作品を自動で作ってくれる機械
- **Azure** = 作品を飾る場所（クラウド上の展示スペース）

## デモの流れ（全7ステップ）

### Phase 0: 環境リセット 🔄

**何をするの？**
前回のデモで散らかったものを片付けます。

**小学生向け説明：**
図工室を使う前に、前の人が使った道具や材料を片付けて、きれいな状態から始められるようにします。

**具体的には：**
- mainブランチ（メインの設計図置き場）に戻る
- 古いブランチ（試作の設計図）を削除
- 未完了のPR（提出してない宿題）を整理
- テスト用のエラーを削除
- 一時保存した変更を削除
- Azureのテスト用リソースを削除

---

### Phase 1: 環境確認 ✅

**何をするの？**
デモを始める前に、必要な道具が揃っているか確認します。

**小学生向け説明：**
図工を始める前に「はさみはあるかな？」「のりはあるかな？」と確認するのと同じです。

**確認すること：**
1. **Azure CLI** - Azureと話すための道具
2. **GitHub CLI** - GitHubと話すための道具
3. **リソースグループ** - 作品を置く場所

---

### Phase 2: リポジトリ・テンプレート・ワークフロー確認 📂

**何をするの？**
設計図とルールを確認します。

**小学生向け説明：**
「今日は何を作るんだっけ？」「どんな手順で作るんだっけ？」を確認します。

**見るもの：**
- **フォルダ構成** - 設計図がどこにあるか
- **Bicepファイル** - ネットワークの設計図（VNet、Subnet、NSGなど）
- **CIワークフロー** - 自動チェックのルール
- **CDワークフロー** - 自動で作る仕組み

---

### Phase 3: ローカルでのBicep検証 🔍

**何をするの？**
自分のパソコンで設計図をチェックします。

**小学生向け説明：**
先生に提出する前に、自分で「書き間違いないかな？」「ちゃんと作れそうかな？」と確認します。

**2つのチェック：**
1. **Bicep Build** - 設計図に間違いがないか確認
2. **What-If Analysis** - 実際に作るとどうなるかシミュレーション

---

### Phase 4: CI失敗デモ ❌➡️📝

**何をするの？**
わざと間違った設計図を提出して、自動チェックが見つけてくれるか確認します。

**小学生向け説明：**
わざと間違った宿題を提出して、先生がちゃんと「これは間違ってるよ！」と教えてくれるか試します。

**流れ：**
1. 新しいブランチ（試作用の設計図）を作る
2. わざと間違いを追加（`resource broken` という壊れた書き方）
3. GitHubに送る（プッシュ）
4. PR（Pull Request）を作る → 「この設計図をメインにしていいですか？」とお願い
5. **CI（自動チェック）が動いて失敗する** ← ここがポイント！
6. GitHub Actionsで「エラーがあります！」と表示される

---

### Phase 5: CI成功デモ ✅

**何をするの？**
間違いを直して、もう一度提出します。

**小学生向け説明：**
先生に「ここが間違ってるよ」と言われたところを直して、もう一度提出します。

**流れ：**
1. 間違いを削除（`resource broken` の行を消す）
2. タイムスタンプのコメントを追加（「いつ直したか」の記録）
3. GitHubに送る（プッシュ）
4. **CI（自動チェック）が動いて成功する** ← ここがポイント！
5. GitHub Actionsで「問題なし！」と表示される

---

### Phase 6: CDデプロイ 🚀

**何をするの？**
チェックが通った設計図を、実際にAzureに作ります。

**小学生向け説明：**
先生のチェックが終わった設計図を、自動で作品にしてくれる機械が動きます。展示スペース（Azure）に作品が飾られます。

**流れ：**
1. CIの完了を待つ
2. PRをマージ（メインの設計図に統合）
3. **CD（自動デプロイ）が動く** ← ここがポイント！
4. Azureにリソースが作られる：
   - **VNet（仮想ネットワーク）** - 大きな敷地
   - **Subnet（サブネット）** - 敷地の中の部屋
     - Web層（外から見える部屋）
     - App層（アプリの部屋）
     - DB層（データベースの部屋）
     - Bastion（管理者専用の入口）
   - **NSG（ネットワークセキュリティグループ）** - 各部屋の鍵と警備員
5. デプロイ結果を確認：
   - リソース一覧
   - VNetの詳細
   - NSGのルール（誰が入っていいか）
   - デプロイ履歴

---

### Phase 7: クリーンアップ 🧹

**何をするの？**
デモで使った試作用の設計図を片付けます。

**小学生向け説明：**
図工が終わったら、試しに描いた下書きは捨てて、机をきれいにします。でも、完成した作品（Azure上のリソース）は残しておくか、全部片付けるかを選べます。

**片付けるもの：**
1. ローカルのデモ用ブランチ
2. 退避した変更（git stash）
3. **Azureのリソース（オプション）** - 全部消すか選べる

## セキュリティの仕組み（多層防御）

### NSGルールを簡単に説明すると...

**学校の警備システムに例えると：**

```
インターネット（外の世界）
    ↓
【正門】Web層のNSG
    ↓ ✅ HTTPとHTTPS（訪問者用の入口）だけ通す
    ↓
【中庭】Web層
    ↓
【中門】App層のNSG
    ↓ ✅ Web層からだけ通す（外の人は入れない）
    ↓
【教室棟】App層
    ↓
【奥の門】DB層のNSG
    ↓ ✅ App層からだけ通す（外の人もWeb層の人も入れない）
    ↓
【金庫室】DB層（一番大事なデータ）
```

**つまり：**
- 外の人は正門（Web層）までしか入れない
- Web層の人は中門（App層）まで入れる
- App層の人だけが金庫室（DB層）に入れる

これを「**多層防御**」と言います。一つの鍵が壊れても、他の鍵があるので安全です！

## GitHub Actionsの仕組み

### CIワークフロー（自動チェック）

**いつ動く？**
- PRを作った時
- Bicepファイルを変更した時

**何をする？**
1. Bicepファイルを読む
2. 間違いがないかチェック（Bicep Build）
3. どんなリソースが作られるか確認（What-If）
4. 結果をPRに表示

### CDワークフロー（自動デプロイ）

**いつ動く？**
- mainブランチにマージされた時
- Bicepファイルが変更された時

**何をする？**
1. Azureにログイン
2. Bicepファイルを使ってリソースを作る
3. 作成結果を表示
4. デプロイの詳細をログに残す

## 重要な概念

### ブランチ戦略

**mainブランチ** = 本番用の設計図
**feature/demo-*ブランチ** = 試作用の設計図

**たとえ：**
- mainブランチ = 先生に提出する清書
- featureブランチ = 試しに書いてみる下書き

### Pull Request（PR）

**PR = 「この設計図を使っていいですか？」というお願い**

**流れ：**
1. 試作ブランチで設計図を作る
2. PRを作る（「これでいいですか？」と聞く）
3. CIが自動でチェック
4. チェックOKならマージ（採用）
5. CDが自動でデプロイ（実際に作る）

## デモで使う主なコマンド

### Git関連

```bash
git checkout -b feature/demo-XXX  # 新しい試作用の場所を作る
git add .                          # 変更をステージング
git commit -m "メッセージ"         # 変更を記録
git push origin feature/demo-XXX   # GitHubに送る
```

### GitHub CLI

```bash
gh pr create     # PRを作る
gh pr merge      # PRをマージする
gh pr list       # PRの一覧を見る
gh run list      # GitHub Actionsの実行履歴を見る
gh run watch     # GitHub Actionsの実行を見守る
```

### Azure CLI

```bash
az login                    # Azureにログイン
az group create             # リソースグループを作る
az bicep build              # Bicepファイルをチェック
az deployment group what-if # 何が作られるかシミュレーション
az deployment group create  # 実際にリソースを作る
```

## よくある質問

### Q1: なんでわざと間違いを入れるの？

**A:** 自動チェック（CI）が本当に動いているか確認するためです。間違いを見つけてくれないと、本番で大変なことになっちゃいます！

### Q2: デモが途中で止まったらどうするの？

**A:** Phase 0の環境リセット機能を使えば、いつでもクリーンな状態に戻せます。

### Q3: Azureにお金かかるの？

**A:** はい、少しかかります。でも、デモが終わったらPhase 7でリソースを削除すれば、料金を止められます。

### Q4: 本番でもこの仕組みを使うの？

**A:** はい！この仕組みがあれば：
- 人間のミスが減る（自動チェックがあるから）
- 何度でも同じものを作れる（設計図があるから）
- 変更の履歴が残る（Gitで管理してるから）
- チーム全員で作業できる（GitHubで共有してるから）

## まとめ

このデモは、こんなことを教えてくれます：

1. **自動化はすごい！** - 人間がやると間違えることも、機械なら正確にできる
2. **チェックは大事！** - 本番に出す前にCIでチェックすれば安心
3. **コードで管理すると便利！** - 設計図をコードで書けば、何度でも同じものが作れる
4. **セキュリティは層で守る！** - 一つの鍵だけじゃなく、何重にも鍵をかける
